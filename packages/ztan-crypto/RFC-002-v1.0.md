# ZTAN-RFC-002 v1.0: Identity Binding & Protocol Authenticity

## 1. Introduction
This document defines the identity registry and authentication requirements for participating nodes in a ZTAN MPC ceremony.

## 2. Node Identity
### 2.1 Public Keys
- Every node MUST register a **48-byte compressed G1 public key**.
- The registry (IdentityService) serves as the source of truth for public key to nodeId mapping.

### 2.2 Lifecycle
- **ACTIVE:** Node is authorized to participate.
- **REVOKED:** Node identity is compromised or decommissioned; contributions MUST be rejected.
- **ROTATED:** Keys may be updated; however, ongoing ceremonies MUST remain bound to the key active at the time of ceremony initialization.

## 3. Protocol Authenticity
### 3.1 Authenticated Messages
All messages in the MPC rounds (commitments, shares, signatures) MUST follow the `AuthenticatedMessage` format:
- `messageId` (UUID)
- `nodeId`
- `payload`
- `signature` (Identity-signed)
- `timestamp`

### 3.2 Idempotency
- The Coordinator MUST track `messageId` in a distributed cache (Redis) and reject duplicates to prevent replay attacks and state corruption.
