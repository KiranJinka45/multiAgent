# ZTAN-RFC-001 v1.5: Audit-Grade Distributed Threshold Signing

**Status**: LOCKED (LATEST)
**Version**: 1.5.0
**Date**: 2026-05-04

> [!IMPORTANT]
> This specification is now IMMUTABLE. Any implementation claiming compliance with ZTAN-RFC-001 v1.5 must follow the binary layout and binding rules defined below without exception.

## 1. Abstract
This specification defines the Zero-Trust Audit Network (ZTAN) distributed signing protocol. It ensures that any cryptographic proof generated by the network is tied to a specific, immutable ceremony configuration, preventing semantic ambiguity and cross-ceremony replay attacks.

## 2. Canonical Encoding (CER v1.5)
All variable-length fields MUST be length-prefixed with a 4-byte big-endian unsigned integer.

### 2.1 Field Encoding
`encodeField(data) = uint32BE(len(data)) || data`

### 2.2 Payload Structure
The audit payload is constructed as follows:
1. `encodeField("ZTAN_CANONICAL_V1")`
2. `encodeField(auditId)`
3. `uint64BE(timestamp)`
4. `encodeField(payloadHash)`
5. `uint32BE(threshold)`
6. `uint32BE(participantCount)`
7. `[encodeField(nodeId)]` (Sorted lexicographically by bytes)

## 3. Cryptographic Binding (RFC v1.5)
To prevent semantic ambiguity, the BLS signature MUST be bound to the full ceremony configuration.

### 3.1 Structured Context
`Context = encodeField(ceremonyId) || uint32BE(threshold) || encodeField(sortedEligiblePublicKeys)`

### 3.2 Binding Hash
`BindingHash = SHA256(encodeField(Context) || encodeField(PayloadHash))`

### 3.3 Signing
The final message signed using BLS12-381 is the `BindingHash`.
**DST**: `BLS_SIG_ZTAN_AUDIT_V1`

## 4. Threat Model
### 4.1 Adversarial assumptions
- The attacker can control up to `t-1` nodes.
- The attacker can observe all network traffic.
- The attacker can attempt to replay signatures across different ceremonies.

### 4.2 Security Guarantees
- **Non-Repudiation**: A valid signature proves that at least `t` distinct nodes from the `eligiblePublicKeys` set agreed to sign the `PayloadHash` under the specific `ceremonyId`.
- **Replay Resistance**: Signatures from Ceremony A cannot be reused in Ceremony B, even if the message hash is identical.
- **Semantic Integrity**: The proof is invalid if the threshold or participant set is altered.
